# 研究 Idea：HySkill — 查询侧假想技能生成用于技能检索与加载

> 状态（2026-07-13 晚）：Phase 0 试点已完成，**GO**（结果见 `phase0-results.md`）。方法形态由试点数据定型：主打单向量想象（三粒度）+ 多金标域四路融合挡位，门控进 Phase 2。
> 目标：顶会/顶刊正式论文。资源：GPU 服务器（A100）本地模型，零 API 依赖。Venue 视进展定。

## 1. 一句话

给定任务，先让 LLM 生成一份"假想 SKILL.md"（内容允许失真），用它的嵌入去技能库检索真技能；并用"假想↔最佳检索结果"的差距做**加载门控**——库里没有合适技能就不硬塞，模型参数化知识足够就不外挂。

## 2. 动机链条（每一环都有已核实的文献支撑）

1. 技能会帮倒忙：库扩到 202 个技能性能降 21%（More Skills, Worse Agents, 2605.24050）；现实检索场景下技能收益逼近零（Skills in the Wild, 2604.04323）。
2. **主因是选错技能（skill shadowing），不是上下文变长**（2605.24050 的条件分解结论）。
3. 选错源于"查询↔技能"的语义鸿沟：技能检索是能力匹配而非语义相似（Skill Is Not Document, 2606.03565）；只看元数据路由准确率暴跌 31–44 pt（SkillRouter, 2603.22455）。
4. HyDE 式假想生成桥接这类鸿沟已被两次验证：文档检索（HyDE, ACL 2023）、API 工具检索（ToolDreamer, EACL 2026 口头）。
5. **技能上的查询侧位置无人占据**（见 background.md 第 5 节逐篇排查）。
6. 额外瓶颈"无差别加载"（SRA, 2604.24594：不管需不需要都加载）与"置信度失准"（Confidence Dichotomy, ACL 2026）为门控贡献提供问题定义。

## 3. 方法设计：三个候选方案

### 方案 1：HySkill-R（纯检索增强，最小可行）

任务查询 → LLM 生成**结构化假想 SKILL.md** → 分字段多视角嵌入 → 检索真技能库（可选轻量重排）。

生成设计要点（区别于 ToolDreamer 的核心设计空间）：
- 生成完整结构：名称 / 描述 / 步骤骨架 / **代码块占位**——SkillFlow 发现 oracle 技能代码块占比显著更高（39% vs 24%），假想技能应带代码骨架而非纯文字；
- 分字段嵌入（元数据、正文、步骤分开）再融合（RRF 或学习权重），对应技能文档的异构结构;
- 生成多份假想技能取向量质心（HyDE 原文即多采样）。

- 优点：最快出结果，预计 2–3 周可有预实验。
- 风险：单独成文易被评"ToolDreamer 的增量迁移"。

### 方案 2：HySkill-RG（检索 + 加载门控）★ 当前推荐

方案 1 + 第二贡献：**门控信号**决定 加载 / 不加载 / 声明库中无合适技能。

候选门控信号：
- sim(假想技能, 最佳检索结果)：差距过大 → 库里没有合适技能，别硬塞（回应 skill shadowing：塞错不如不塞）；
- 假想技能的"自足性"：模型自己就能写出完整可执行的假想技能 → 参数化知识足够，无需外挂（回应 MESA-S 的自信/源信区分）；
- 两信号可组合成三区间决策（加载 / 跳过 / 报告无合适技能）。

评测不只报检索指标，还报端到端任务通过率与 **skill shadowing 率的下降**。

- 优点：从"检索技巧"升级为"检索 + 元认知门控"的完整故事；直接回应两个已被顶会/基准确认的瓶颈；新颖性防御强；工作量适中。

### 方案 3：HySkill-RGF（+ 融合精炼）

门控放行后，以假想技能为模板把真技能**按当前任务改写融合**（真技能出事实与代码，假想技能出任务对齐的结构）。接上 Skills in the Wild 已验证有效的"查询级精炼"。

- 优点：三段贡献，天花板最高。
- 风险：范围膨胀，精炼单独就是一篇论文的量；时效窗口可能等不起。
- 处置建议：精炼作为方案 2 的消融/扩展章节，时间不够则留 future work。

### 方案节结局（2026-07-13 试点后）

实际路线在推进中演化为：**方案 2 的骨架 + 试点驱动的方法定型**。检索部分不再是单一方法，而是"想象粒度 ×（单向量 / 四路融合）"的变体族，由试点数据裁决出双挡位规律（见 3.5 节与 `phase0-results.md`）；门控（RG 的 G）按计划进 Phase 2；精炼（RGF 的 F）维持 future work。

## 3.5 创新点的三层结构（论文定位主轴，2026-07-13 补）

1. **对象层**：查询侧假想生成首次进入技能领域——连 naive_hyde 基线的数字都是领域内全新结果（无人发表过 HyDE-on-skills）；
2. **方法层（试点后修订，2026-07-13）**：**双挡位 + 挂挡规律**。单路单向量（信号干净）与分字段四路融合（广撒网）各擅胜场，试点五域实测：**单金标域（theoremqa/logicbench/medcalc）单路全胜，多金标域（champ/bigcodebench）融合全胜**——"映射方式应匹配域的金标结构"是可检验的规律，Phase 1 目标是把它做成自适应方法（按查询预测单/多书结构自动挂挡，或双挡并行由门控信号裁决）；
3. **加载层（最独家）**：假想文档第一次被用到检索之后的环节——假想技能兼任"参数化知识外化快照"，驱动 S1/S2 加载门控，直击 SRA 证实的"无差别加载"瓶颈。HyDE 与 ToolDreamer 均止步于检索。

战略保险：三层独立成立；试点已证实对象层（五域全冠军，平均 nDCG@10 超最强基线 47%）。检索实验的生成缓存直接复用为门控实验的输入。

## 4. 新颖性防御（预答审稿人）

| 潜在质疑 | 防御 |
|---|---|
| "ToolDreamer 已做过 HyDE→工具" | 对象本质不同：API 工具是几十 token 类型化 schema，技能是数百至数千 token 程序性异构文档；"生成什么样的假想技能"（结构、代码骨架、分字段嵌入）是 ToolDreamer 中不存在的设计空间 |
| "SkillDAG 已把 HyDE 引入技能" | 方向正交：SkillDAG 仅库侧离线索引（e_needs，逐技能、不感知任务），推理时查询原样嵌入；本工作是查询侧在线任务条件化生成。且 SkillDAG 可作**消融宿主**（把查询侧假想技能插进其流水线展示即插即用增益），双向结合（查询侧 × e_needs）为免费扩展实验 |
| "SkillFlow 已做查询生成" | 显式搜索查询 ≠ 假想文档嵌入；SkillFlow 论文自己与 HyDE 划界且未测 HyDE 基线 |
| "门控是拍脑袋" | 问题定义借 SRA（无差别加载）与 Confidence Dichotomy（ACL 2026，置信度失准）；信号有明确假设与消融 |

## 5. 评测计划

**基准**（全部现成，无需自建）：SRA-Bench（26,262 技能，全流程）、SkillsBench（端到端）、Skill-Is-Not-Document 基准（查询条件化检索）、Terminal-Bench（端到端，注意 SkillFlow 在此无显著增益的前车之鉴）、可选 CompSkillBench（组合场景）。

**基线**：BM25 / 稠密（BGE、Contriever）/ 混合 / LLM 重排（SRA-Bench 原装四件套）+ 朴素 HyDE 直接移植（不做技能结构化，证明结构化生成的必要性）+ ToolDreamer 移植 + SkillFlow + SkillRouter + SkillDAG。

**指标**：检索层 Recall@K / Hit@1；端到端 Pass@1、token 成本、交互步数；机制层 skill shadowing 率（借 2605.24050 的事件分类法）；门控层 precision/recall（该加载时加载、不该时拒绝）。

**模型**：生成器与执行器解耦评测（小模型生成假想技能 + 大模型执行 的成本敏感配置）；GPT-5.x / Claude / Qwen 开源系（开源系供门控信号的白盒校准分析，用 GPU）。

## 6. 风险与时效

- **撞车风险（最高优先）**：SkillDAG v2 2026-07-02 刚更新，赛道 3 个月约 10 篇论文。对策：先跑最小预实验（SRA-Bench 上 HySkill-R vs 原装基线），结果为正即挂 arXiv 占位，再扩门控。
- 负结果风险：若假想技能检索不涨点，检查生成质量/嵌入模型适配（HyDE 对编码器敏感）；门控作为独立贡献可单独成立。
- 工程风险低：无需训练检索器（零样本路线），基准与基线代码多已开源。
