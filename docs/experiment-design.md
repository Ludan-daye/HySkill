# 实验思路：参照 SRA-Bench 的设计与我们的改造

> 2026-07-13。配套文档：方法规格见 `docs/superpowers/specs/2026-07-13-hyskill-design.md`，实现计划见 `docs/superpowers/plans/2026-07-13-hyskill-phase0.md`，HyDE 原文精读见 `docs/hyde-method.md`。

## 1. 参照实验：SRA 论文是怎么做的

**他们想回答的问题**：给 AI 一个巨大的"技能说明书库"，它到底会不会用？

**实验搭建（图书馆比喻）**：出 5,400 道题（定理应用、逻辑推理、工具流、医学计算、数学概念、代码库使用六类）。每道题在图书馆里都有一本能帮上忙的"正确参考书"——这 636 本金标技能是作者手写的；再混入 25,626 本网上收集的干扰书，共 26,262 本，让找书变成大海捞针。

**核心巧思：把"用书答题"拆成三步，每步单独打分**（官方 CLI 即三条命令）：

| 阶段 | 命令 | 测什么 | 打分 |
|---|---|---|---|
| ① 找书（检索） | `sragents retrieve` | 搜索系统能否把正确书排到前面 | Recall@K / nDCG@K |
| ② 决定（纳入） | `sragents infer` | AI 看到候选后自己决定拿不拿书 | Skill Loading Rate + 两个诊断维度（见下） |
| ③ 做题（应用） | `sragents evaluate` | 照书做题答案对不对 | accuracy / pass@1 |

第②步的聪明对照：故意分成"候选里**有**正确书"和"**没有**正确书"两种情况，看 AI 拿书频率变不变（**relevance-aware**）；再看任务超出 AI 自身能力时它是否更倾向拿书（**need-aware**）。

**为什么拆三步**：只看最终答案无法定位失败——是没找到书、拿错了书、还是拿对了没看懂。拆开才能定位病灶。

**他们的大发现**：找书还凑合，坏在"决定"——**大多数模型拿书像抽奖**：候选里有无正确书，加载率几乎不变（Qwen3-4B：21.8% vs 16.9%；Llama-70B 甚至倒挂）。他们称之为"技能加载幻觉"（skill-loading hallucination）。**但他们只诊断了病，没开药。**

模型矩阵：Qwen3（4B/32B/235B）、Llama-3.1-8B / 3.3-70B、Mistral-24B，另有 GLM-5.1、GPT-5.4 行为分析。

## 2. 我们借鉴什么、为什么

**全部借用**：26,262 技能语料、5,400 实例及金标标注、三段 CLI 流水线、全部指标代码。方式是官方插件机制接入（`sragents --plugin hyskill.plugin`），**SR-Agents 源码一行不改**。

三个理由：
1. 自建基准会被审稿人质疑，用他们的没人能挑刺；
2. 结果与他们论文直接可比；
3. 省数月工作量——他们的"skill-free correct/wrong"标注（不用技能能不能做对）正好是我们门控阈值校准的现成真值。

## 3. 我们的实验一：找书比赛（Phase 0，检索层）

同一批题、同一个图书馆，五个选手同场竞技：

| 选手 | 找书方式 | 身份 |
|---|---|---|
| BM25 | 查字面词 | 原装基线 |
| bge | 题目变"语义指纹"直接找 | 原装基线 |
| 混合 | 上两者结合 | 原装基线 |
| naive_hyde | 让 AI 先编一段假答案，用假答案的指纹找 | 我们移植的 HyDE 原版 |
| **hyskill** | 让 AI 先**编一本假技能说明书**（名字/步骤/代码骨架），封面找封面、正文找正文、代码找代码，四路汇总 | **我们的方法** |

**公平性（归因设计，抄自 HyDE 原文）**：bge / naive_hyde / hyskill 共用同一个指纹模型和同一个图书馆索引，**唯一区别是拿什么去找**。结果自动拆成两级归因：

- naive_hyde − bge = "想象"这个点子本身值几分；
- hyskill − naive_hyde = "把想象写成一本正经的技能书"（结构化 + 分字段 + 融合）额外值几分。

**判据（go/no-go）**：hyskill 在多数数据集显著最好 → 推进 + 挂 arXiv 占位；hyskill ≈ naive_hyde → 结构化设计没用，回炉改生成模板，不烧大钱。

## 4. 我们的实验二：替 AI 做决定（Phase 2，门控层）

他们证明 AI 自己决定拿不拿书是乱来的，我们把这个决定换成两个**算出来**的信号（纯向量运算，零额外 LLM 调用）：

- **"像不像"（S1 覆盖）**：想象中的书 vs 找到的最像的书差多远。差太远 = 馆里没这书，硬塞错书只会带偏 → 不拿。
- **"有没有新货"（S2 增益）**：找到的书里有多少内容是 AI 自己想象不出来的。逻辑：**AI 能想象出来的内容就是它本来会的**；书里的东西它全能自己写 → 拿书纯属浪费 → 不拿。

**验收标准直接用他们的尺子**：报他们定义的 relevance-aware / need-aware 分离度改善 + 端到端 accuracy 变化，与他们原装三种暴露策略（全量注入 / LLM 选择 / 渐进披露）对比——用他们的诊断指标证明我们治好了他们诊断出的病。

## 5. 刻意不照搬的部分

他们的技能偏知识型（定理、计算器、逻辑模式），不全是 SKILL.md 程序型技能。所以 Phase 3 端到端换到 SkillsBench / Terminal-Bench 补程序型场景，避免结论被"技能类型"限定。ToolQA 需外部语料，Phase 0 暂缓。

## 6. 节奏与预算

| 步骤 | 规模 | 成本 | 目的 |
|---|---|---|---|
| 试点 | 每数据集 20 实例 | 几美元 | 验流程 + 人工检查 AI 编的假技能书像不像样 |
| 全量 Phase 0 | 5 数据集全部实例 | 约 6.5M token 生成，数十美元 | go/no-go 正式数字 |
| Phase 1 消融 | 按需 | 生成缓存后只付增量 | K、生成器规模、逐路贡献、查询锚、编码器 |

**已有真实数字**（2026-07-13，theoremqa 前 20 实例 × 完整 2.6 万语料）：
BM25 基线 Recall@1=0.600，Recall@10=0.800，nDCG@10=0.712——这是要超过的第一道线。

一句话总结：**他们把"用技能"拆成找、决定、用三步并发现 AI 坏在"决定"；我们借他们的整套考场，在"找"这步插入想象力（假想技能生成），在"决定"这步插入两个替 AI 拿主意的信号，每一步都用他们原装的评分标准验收。**
