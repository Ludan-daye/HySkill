## ① Idea：研究想法的来源与三层创新点

### 1.1 一句话

给定任务，先让 LLM 生成一份"假想 SKILL.md"（内容允许失真），用它的嵌入去技能库检索真技能；再用"假想 ↔ 最佳检索结果"的差距做**加载门控**——库里没有合适技能就不硬塞，模型参数化知识足够就不外挂。（`06-idea-and-decisions.md` §1）

### 1.2 想法来源（动机链条，每一环有已核实文献，见 `01-background.md`）

1. 技能会帮倒忙：技能库扩到 202 个技能，性能最多降 21%（More Skills, Worse Agents, 2605.24050）；从 34,000 个真实技能中自行检索时，最难场景下技能收益逼近零（Skills in the Wild, 2604.04323）。
2. 主因是**选错技能（skill shadowing）**，不是上下文变长（2605.24050 的条件分解结论）。
3. 选错源于"查询 ↔ 技能"的语义鸿沟：技能检索是能力匹配而非语义相似（Skill Is Not Document, 2606.03565）；只看元数据路由准确率暴跌 31–44 pt（SkillRouter, 2603.22455）。
4. HyDE 式查询侧假想生成桥接这类鸿沟已被两次验证：普通文档检索（HyDE, ACL 2023）、API 工具检索（ToolDreamer, EACL 2026 口头）。
5. 逐篇排查（`01-background.md` §5.3，2026-07-13 核实）："任务查询 → 生成假想技能文档 → 检索真技能"这一组合位置当时无人做过。
6. 另一瓶颈"无差别加载"（SRA-Bench, 2604.24594：候选里有无正确技能，模型加载率几乎不变，Qwen3-4B 21.8% vs 16.9%）为门控提供了问题定义——SRA 只诊断了病，没开药。

### 1.3 与最近邻工作的事实差异（新颖性定位）

| 最近邻 | 它做了什么 | 与本项目的差异 |
|---|---|---|
| HyDE（ACL 2023） | 查询侧假想文档 → 检索**普通文档** | 对象不同：技能是数百~千 token 的程序性异构能力包（含步骤与代码） |
| ToolDreamer（EACL 2026 口头） | 查询侧假想**工具描述** → 检索 API 工具 | 对象不同：API schema 是几十 token 的类型化描述；且止步于检索环节 |
| SkillDAG（arXiv 2606.03056） | **库侧**索引期为每技能生成 e_needs 摘要（引 HyDE）；推理时查询原样嵌入 | 方向正交：离线逐技能、不感知当前任务；本项目是查询侧在线任务条件化生成 |
| SkillFlow（arXiv 2504.06188） | 查询侧生成**显式搜索查询**（论文自我声明非 HyDE，无 HyDE 基线） | 生成搜索词 ≠ 生成假想文档嵌入 |
| SRA-Bench llm_rerank / LLM 选择 | 检索后让 LLM 读候选原文重排/自选（SRA 报告的最强基线） | 不生成假想文档；每题需额外 LLM 调用且受 BM25 召回天花板锁死（Phase 1/2 中同场直测对比） |

HyDE 谱系（HyDE→ToolDreamer→SkillDAG）全部止步于检索环节；把假想文档用到检索**之后**的加载决策，此前没有先例（`02-innovations.md` 创新点 3）。

### 1.4 三层创新点（终版表述）

1. **对象层**：查询侧假想生成首次进入技能（SKILL.md）领域——连 naive_hyde 基线的数字都是该领域内的全新结果。
2. **方法层**：想象不是单一技巧，而是一个设计空间——**粒度轴**（一句话 / 一段话 / 完整技能）×**映射轴**（单向量 / 分字段四路融合），加两阶段架构共 5 个变体；Phase 2c 进一步加入**验证集粒度路由**：用 20% 验证集为每个域自动选变体（离线回放 5/5 命中全量冠军）。
3. **加载层**：假想技能兼任"模型参数化知识的外化快照"——模型能自己写出来的内容就是它已经会的。据此计算 S1（覆盖）/S2（增益）两个信号替模型做加载决策，**零新增 LLM 调用**；Phase 2/2c 端到端做题实验证实门控四域全不降、路由+门控合并 72.5% 为可部署臂第一（数字见 ④）。

### 1.5 演化过程要点（决策记录见 `06-idea-and-decisions.md` §7）

- 2026-07-13：由"复杂技能导致性能退化（机制+缓解）"选题**全面转向**本 idea；完成新颖性排查。
- 方法候选有三档（纯检索 R / 检索+门控 RG / 加融合精炼 RGF），实际路线为 RG 骨架；精炼（F）保留为未来工作。
- 方法主线曾定为单向量想象、四路融合降级为变体（判断依据：真实技能库异构、字段映射脆弱）；试点后四路在多金标域翻身，形成"双挡位"并催生两阶段架构。
- Phase 0 试点（每域 20 题）判定 GO：想象类五域全冠军——但全量后部分翻转为域冠军 3:2，试点教训被明确记录："20 题只能筛方向，不能排座次"（`05-results.md` 附录 A）。
- Phase 2c 前曾评估逐题路由的三种方案，全部被离线回放否决（逐题裁判数学退化为恒选 naive_skill；难度阈值滑向极端退化为域级），最终采用域级验证集路由（`docs/superpowers/specs/2026-07-14-multimodel-v2-design.md`）。

## ② 框架：方法与系统架构

### 2.1 流水线总览

```
任务查询 q
  → 想象生成（3 模板 × K=4 采样，温度 0.7，内容寻址磁盘缓存）     [hyskill/generator.py]
  → 嵌入（MiniLM all-MiniLM-L6-v2；语料嵌入磁盘缓存）             [hyskill/embedder.py]
  → 检索（5 变体之一，见 2.2）                                    [naive_hyde/retriever/two_stage.py]
  → （Phase 2c）域级粒度路由：验证集选变体                         [scripts/route_variant.py]
  → S1/S2 加载门控：放行 → 注入 top-1 技能做题；拦截 → 裸考        [scripts/gate.py]
```

接入方式：SR-Agents 官方插件机制（`sragents --plugin hyskill.plugin`），SR-Agents 源码一行不改（`hyskill/plugin.py`；同时注册 dense、fast_bm25，并在运行时把 llm_rerank 的 max_tokens 封顶 1024）。

### 2.2 想象生成与 5 个检索变体（粒度轴 × 映射轴）

想象生成：每题按模板生成 K=4 份假想文档，温度 0.7；三种模板对应三档粒度——一句话（技能名+一句功能，~30 token）、一段话（HyDE 式答案短文，~150 token）、完整技能（仿真 SKILL.md，含名称/描述/步骤/代码骨架，~300 token）。多份取向量质心，并按 HyDE 公式 8 混入"查询锚"（查询自身用文档编码器编码，作为第 K+1 份假设）。思考型模型（Qwen3/3.5）需 no_think 关思考模式，外层 markdown 围栏由解析器剥壳。

| 变体 | 粒度 | 映射 | 模块 |
|---|---|---|---|
| 想象·一句话（naive_sentence） | 一句话 | 单向量 | `hyskill/naive_hyde.py` |
| 想象·一段话（naive_passage） | 一段话 | 单向量 | `hyskill/naive_hyde.py` |
| 想象·完整技能（naive_skill） | 完整技能 | 单向量 | `hyskill/naive_hyde.py` |
| 想象·四路融合（hyskill） | 完整技能（同一份想象） | meta/body/code 字段对字段 + BM25，RRF（k=60）合并 | `hyskill/retriever.py` + `parser.py` + `fusion.py` |
| 两阶段（two_stage） | 完整技能 | 四路融合召回 → 单向量精排 | `hyskill/two_stage.py` |

注意：四路不是第四种想象——与完整技能共用同一份想象，差异仅在映射端。BM25 后端为 bm25s（method=robertson，与 rank_bm25 排序一致、已单测锁定，快约百倍）+ rank_bm25 兜底（`hyskill/bm25.py`）。

### 2.3 验证集粒度路由（L1，Phase 2c）

与门控共用同一个 20% 验证集（sorted ids 上 `random.Random(0).sample`，seed 0）；每域在验证集上算 5 个变体的平均 nDCG@10 取 argmax，产出该域的路由检索文件；测试报告只用其余 80%。离线回放中 5/5 域命中全量冠军。

### 2.4 S1/S2 加载门控与保守标定

- **S1 覆盖** = cos(想象质心+查询锚, top-1 技能全文)：过低 → 库中无合适技能，不加载（防 skill shadowing）；
- **S2 增益** = top-1 中想象覆盖不到的句子占比：过低 → 模型已会，跳过（防无差别加载）。

全部信号复用检索阶段已缓存的想象文档——门控阶段 **0 次新 LLM 调用**。阈值在 20% 验证集上保守标定（拦截精度 ≥90%）；路由后门控信号与阈值对新 top-1 **重新计算与标定**，但 S1/S2 始终用完整技能想象（知识快照角色），路由只改排序来源。

### 2.5 缓存复用："想象一次，用三次"

同一批想象文档依次复用于**排序（检索）/ 路由 / 门控**三层：Phase 1 全量缓存预热 47,616 份生成（0 失败），Phase 2/2c 的信号与路由全部从缓存读取，未新增生成；全项目累计约 6.3 万份生成横跨全部实验零重付。语料嵌入同样磁盘缓存（26,262 技能矩阵 CPU 编码约 27 分钟 → A100 约 1 分钟，缓存是全量实验的硬前提）。

### 2.6 工程配套

`scripts/`：warm_cache.py（并行想象缓存预热）、gate.py（S1/S2 门控+标定）、route_variant.py（L1 路由）、significance.py / phase2_significance.py（检索/做题配对 bootstrap 显著性）、analyze.py / summarize_multimodel.py（汇总；后者含实测成本审计块）、run_phase0.sh / run_phase2a-c.sh / run_multimodel.sh（批跑，断点续跑）、smoke.sh（离线 mock 冒烟）。测试：41 个单元测试 + 微型语料（`tests/`）。全程本地模型（Qwen3.5-4B vLLM + MiniLM），API 费 0。

### 4.3 实测成本账（token 计量，Qwen3.5-4B）

想象·完整技能 K=4 每题 1,760 入 + 976 出 ≈ **2,736 tokens**；重排每题 ~3,700 入（50 候选长 prompt）+ ~200 出 ≈ **3,900 tokens**。**单次在线成本同数量级**（纯 FLOPs 口径想象省 1.4 倍；按"输出贵 3-4 倍"的 API/墙钟口径大致打平）。早期"约 10 倍成本差"的说法已被此实测账**废止**，不再使用；正确口径是"**同等在线成本下（路由后）显著更准**"。想象的结构性优势在别处：①同一批想象 token 复用于检索/路由/门控三层（本项目 6.3 万份生成横跨全部实验零重付，摊销后 ≥5 倍）；②只依赖题目、与库解耦——库更新重排缓存全作废，想象缓存不动；③可离线预生成；④K 消融（进行中）若 K=1–2 保精度，在线成本再砍至重排的 1/3–1/6。另：重排受 BM25 召回天花板锁死（medcalc 上 0.898 < 想象 0.995 即因 R@50=0.925 封顶）。

成本审计已固化进管线：`summarize_multimodel.py` 从每次运行的真实产物实测各方法 token 成本（估算器统一 chars/3.8），写入 summary.json 的 `cost` 块；规则是任何进入对比的方法必须携带同批次实测成本（`08-multimodel-plan.md` §6）。

| **gated（我方）** | 68.0 | 70.7 | **78.2** | 45.7 | **70.9** |

| oracle（天花板） | 74.0 | 84.2 | 79.6 | 56.5 | 77.6 |

标定出的门（同一套规则自动适应四个域）：theoremqa τ₁=0.429 拦 10/747，logicbench τ₁=0.612 拦 751/760，medcalc τ₁=0.606 拦 4/1100，champ τ₁=0.438 拦 3/223；S2 跳过门全程零触发（保守标定生效）。

**三个预注册问题的答案**（配对 bootstrap，10,000 重采样）：

- **Q1 无差别加载伤不伤？——伤，且被均值掩盖。** always−bare 合并 +6.1（p<0.001），但拆域：medcalc +15.1（p<0.001）、theoremqa +7.0（p<0.001）是红利域；logicbench −4.6（p=0.012）、champ −4.9（p=0.092）是遮蔽域。oracle 在 logicbench 高达 84.2（比裸考 +12.1）说明库里有好书、是检索找不到：遮蔽 = 检索失败 × 无差别注入。
- **Q2 门控救得回来吗？——四域全部不降反升，合并 +1.34（p=0.066）。** gated−always：theoremqa +0.3、logicbench +3.2、medcalc +0.7、champ +1.8。对裸考合并 +7.4（p<0.001），是三条可部署臂的第一名。边界：logicbench 救回伤害的 ~70%，champ 只救回 ~36%（S1 抓得住"库里没对口书"，抓不住"题材对口但误导"的书）。
- **Q3 外部门控 vs 模型自觉（select）？——准确率统计打平（合并 +0.4，p=0.65），成本差一个数量级。** 分域 2:2：select 在 theoremqa（+2.0，n.s.）和 champ（+5.8，p=0.047）占优；gated 在 logicbench（+1.2）和 medcalc（+2.7，p=0.014）占优（medcalc 上 select 75.5 连 always 77.5 都不如）。代价：select 全程多 5,660 次推理；gated 零新调用、信号在 GPU 上分钟级算完。

### 4.5 Phase 2c：粒度路由 + 门控（v2 完整流水线）

路由协议：与门控共用同一 20% 验证集（seed 0），每域按验证集 nDCG@10 在 5 个想象变体中选冠军（离线回放 5/5 命中全量冠军）；门控信号与阈值对路由后的新 top-1 重新计算与标定。实跑路由结果：theoremqa→两阶段、logicbench→四路、medcalc→一段话、champ→两阶段；门控拦截量自适应为 9 / 431 / 0 / 3。

**检索端 head-to-head（同一测试集，n=3,177 配对）**：路由想象 nDCG@10 合并 **0.687 vs LLM 重排 0.662（+2.5，95% CI [+1.3, +3.7]，p<0.0001）**，宏平均 0.636 vs 0.607——同等在线成本下（见 4.3）显著更准。域冠军 3:2（想象：theoremqa/medcalc/champ；重排：logicbench/bigcode）。

**做题端七臂终表（合并 2,830 题，准确率 %）**：

| 臂 | theoremqa | logicbench | medcalc | champ | 合并 |
|---|---|---|---|---|---|
| bare | 60.8 | 72.1 | 62.4 | 48.9 | 63.5 |
| always（固定） | 67.7 | 67.5 | 77.5 | 44.0 | 69.6 |

### 4.6 全部规律（1–9，照录 `05-results.md`）

- **规律 1（主结论）**：查询侧假想生成显著优于全部传统基线（五域全部 p<0.001，最强 hybrid 均值 0.456 vs 想象系 0.58–0.60）；在其擅长的域对 LLM 重排也显著取胜（theoremqa +0.068、medcalc +0.097，均 p<0.0001）。
- **规律 2（想象 vs 重排的势力版图，终稿）**：域冠军 3:2（想象系 2 显著 + 1 n.s. vs 重排 1 显著 + 1 n.s.），五域均值几乎打平（0.604 vs 0.613）；粒度路由后合并**显著反超**（§5.4）。单次在线成本同数量级（实测账见 §2），想象的优势是结构性的：token 一批三用（检索/路由/门控）、与库解耦可离线缓存、不受 BM25 召回天花板锁死（medcalc 实证）。重排的相对优势集中在"嵌入信号弱、字面信号尚可"的域（logicbench/bigcode）。两者组合（想象召回 + LLM 精排）是自然的未来工作。
- **规律 3（两阶段的适用边界）**：强信号域登顶（theoremqa 显著、champ 并列、medcalc 把融合的 0.824 修复到 0.981），难域反而拖累纯融合（logicbench 0.307 vs 0.368、bigcode 0.467 vs 0.575）——**想象信号可靠时精排增益，不可靠时破坏融合原序**。
- **规律 4（粒度 × 域）**：一句话在代码域塌方（0.198）、强信号域三档粒度几乎无差（medcalc 均 >0.98）——粒度只在中等难度域重要；完整技能仍是均值最高且最稳的单路形态（0.594）。
- **规律 5（遮蔽实锤）**：无差别加载在 4 域中 2 域净伤害（logicbench −4.6 显著、champ −4.9 边缘），伤害被跨域均值掩盖——评加载策略必须分域看。
- **规律 6（门控 = 检索失败的保险丝）**：想象↔现实差距（S1）驱动的门控四域全不降、合并 +1.34/+7.4（对 always/bare），同一保守标定自动适应库质量（拦截率 1.3%–98.8%），零新增推理。
- **规律 7（几何门与语义门互补）**：S1 抓"库里没书"（logicbench 型失败），模型自选抓"有书但误导"（champ 型失败）；两者统计打平但成本差一个量级——便宜的先上门控，贵的自选留给高价值请求，或做级联（未来工作）。
- **规律 8（路由的做题红利与边界）**：champ 被路由"治好"——它并非库有毒而是检索不行,换两阶段后塞书从 −4.9 转为 **+3.6**（对裸考）,连 select 也被超过；medcalc 路由+门控（81.6）名义越过 oracle（79.6,温度 0.7 波动内=触顶）。但 logicbench 路由单用**有毒**（−2.1）：更"像"的错书更能误导——**检索指标涨 ≠ 做题涨,遮蔽域中代理指标与端到端效果脱钩**。
- **规律 9（油门+刹车缺一不可）**：路由（油门）在 champ/medcalc 大赚但 logicbench 翻车；门控（刹车）在 logicbench 救场但不产生红利。合体 gated_r 无一域落败,合并 72.5% 显著优于所有可部署对手——包括每题多付一倍推理的模型自选。

## ⑤ 诚实边界与进行中的工作

**已知边界（如实记录）**：

- **champ 型漏检**：S1 门控抓得住"库里没对口书"，抓不住"题材对口但误导"的书——champ 竞赛题的近邻技能题材相似但解法误导，S1 偏高不拦；该域伤害只救回约 36%（logicbench 约 70%）。

- **检索指标与做题效果在遮蔽域脱钩**（规律 8）：logicbench 上路由单用使做题下降（−2.1），必须配门控。
- **离 oracle 仍有距离**：最优可部署臂 gated_r 合并 72.5 vs oracle 77.6。
- **K 消融未做**：K∈{1,2,4,8}（K=8 备料重拉中）；生成器规模（Qwen3.5-9B）与编码器（bge）消融同样待做；目前全部结果出自 K=4 + MiniLM。
- **bigcodebench Stage 3 端到端未跑**（需安装 django 等执行依赖）；做题结论目前基于 4 个规则可评域的知识型技能，程序型技能场景（SkillsBench / Terminal-Bench）未测。

- **单一生成模型**：现有全部结果出自 Qwen3.5-4B。
